Visión general

ControlBox es una plataforma SaaS de hosting multi-tenant que permite a proveedores de infraestructura ofrecer gestión de sitios web, bases de datos, DNS, FTP, backups, WordPress, entornos staging y más, desde un panel unificado.

⚡

Backend

FastAPI + PostgreSQL + Redis + Celery. API REST en /api/v1.

🖥️

Frontend

Next.js 15 con panel de administración responsive y autenticación por cookies HttpOnly.

🐳

Infraestructura

Docker, Traefik (SSL automático), Nginx, provisionamiento por contenedor por sitio.

Instalación en producción

Método recomendado para servidores dedicados o VPS. Instalador remoto con configuración automática de firewall, SSL y servicios.

Opción A — Instalación remota (CDN)

curl -fsSL https://install.grodtech.com/install.sh | bash

Opción B — Instalación no interactiva (dominio + tenant)

curl -fsSL https://install.grodtech.com/install.sh | \
  CONTROLBOX_PRIMARY_DOMAIN=tudominio.com \
  CONTROLBOX_ADMIN_EMAIL=admin@tudominio.com \
  CONTROLBOX_TENANT_NAME="Mi Empresa" \
  CONTROLBOX_TENANT_SLUG=mi-empresa \
  CONTROLBOX_TENANT_ADMIN_EMAIL=admin@mi-empresa.com \
  CONTROLBOX_TENANT_ADMIN_PASSWORD='ContraseñaSegura123!' \
  CONTROLBOX_TENANT_ADMIN_FULL_NAME="Administrador" \
  CONTROLBOX_PANEL_PORT=8475 \
  CONTROLBOX_ASSUME_YES=true \
  bash

Opción C — Instalación local desde el repositorio

cd installer
sudo CONTROLBOX_INSTALLER_ROOT=$(pwd) bash install.sh

Asistente interactivo

Al ejecutar el instalador sin variables, se muestra un asistente que solicita:

  1. Dominio principal y email SSL (opcional; puede configurarse después)
  2. Datos del primer tenant: organización, slug, email, contraseña y nombre del administrador
  3. Puerto del panel en el VPS (se asigna uno aleatorio entre 8000–9999 si no se indica)

Qué instala el script

  1. Verifica requisitos de OS, RAM, CPU y disco
  2. Recopila la configuración del tenant y del panel
  3. Instala Docker y Docker Compose si no están presentes
  4. Genera secretos con openssl rand en /etc/controlbox/platform.env
  5. Configura firewall (UFW o firewalld): SSH, 80, 443 y puerto del panel
  6. Despliega stack Docker: PostgreSQL, Redis, Traefik, MinIO, API, Panel, Prometheus, Grafana, Loki, Supabase
  7. Ejecuta migraciones Alembic (alembic upgrade head)
  8. Crea automáticamente el primer tenant (sin habilitar REGISTRATION_ENABLED)
  9. Configura backup automático diario a las 03:00 (retención 14 días)
  10. Muestra credenciales de acceso al panel (URL, usuario y contraseña)
  11. Crea snapshots de rollback en /var/lib/controlbox/state/rollback/
Rollback automático Si la instalación falla, se restaura el snapshot anterior. Manual: sudo bash /opt/controlbox/repair.sh --rollback

Instalación con Docker Compose (desarrollo)

Para entornos de desarrollo o pruebas locales en el repositorio del proyecto.

  1. Clonar el repositorio y copiar variables de entorno: 
    cp .env.example .env
  2. Editar .env y configurar obligatoriamente:
    • APP_SECRET_KEY — cadena aleatoria de 64+ caracteres
    • POSTGRES_PASSWORD
    • REDIS_PASSWORD
  3. Levantar servicios core: 
    docker compose up -d
  4. Ejecutar migraciones: 
    docker compose --profile migrate run --rm migrate
  5. (Opcional) Activar perfiles adicionales: 
    docker compose --profile databases up -d
docker compose --profile dns up -d
docker compose --profile ftp up -d
docker compose --profile backups up -d
docker compose --profile security up -d

Post-instalación

Credenciales de acceso

Al finalizar la instalación, el instalador muestra un resumen con la URL del panel, usuario y contraseña:

URL PANEL:     75.148.75.12:8475/ControlBox_Panel
USUARIO:       admin@mi-empresa.com
CONTRASEÑA:    ContraseñaSegura123!

Las mismas credenciales se guardan en /etc/controlbox/credentials.txt (permisos 600).

El panel usa la ruta base /ControlBox_Panel y un puerto dedicado en el VPS para acceso directo por IP antes de configurar dominios.

Configuración básica desde el panel

Tras iniciar sesión, el usuario Owner o Administrator solo debe completar la configuración básica en Settings:

  1. Producción — checklist y rotación de secretos de instalación
  2. Acceso al panel — cambiar puerto y ruta (/ControlBox_Panel)
  3. Alertas — umbrales de CPU, RAM y almacenamiento (notificaciones en tiempo real)
  4. Seguridad — activar TOTP/MFA para administradores
  5. Dominios — controlbox domains tudominio.com admin@email.com

Si cambia la ruta del panel desde Settings, ejecute en el servidor:

sudo controlbox apply-panel
# o: sudo bash /opt/controlbox/repair.sh --apply-panel

Primer tenant (cuenta administradora)

Automático en producción El instalador crea el primer tenant durante la instalación. No es necesario habilitar REGISTRATION_ENABLED ni usar curl manualmente.

El asistente de instalación solicita los datos del administrador y ejecuta un bootstrap interno (controlbox.installer.bootstrap_tenant) con un token de instalación de un solo uso (INSTALLER_BOOTSTRAP_TOKEN). El registro público permanece deshabilitado (REGISTRATION_ENABLED=false).

El registro crea automáticamente: tenant, usuario admin, todos los permisos RBAC, rol owner en Team Members y sesión inicial.

Creación manual (solo desarrollo o recuperación)

Si necesita crear un tenant adicional en desarrollo, puede habilitar temporalmente el registro público:

  1. En .env (dev), establecer: 
    REGISTRATION_ENABLED=true
  2. Reiniciar la API: 
    docker compose restart api
  3. Registrar el tenant vía API: 
    curl -X POST http://localhost:8000/api/v1/identity/tenants/register \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Mi Empresa",
    "slug": "mi-empresa",
    "admin_email": "admin@mi-empresa.com",
    "admin_password": "ContraseñaSegura123!",
    "admin_full_name": "Administrador"
  }'
  4. Deshabilitar registro público nuevamente: 
    REGISTRATION_ENABLED=false

Módulo de seguridad

MFA (TOTP)

Autenticación de dos factores compatible con Google Authenticator, Authy, etc.

Passkeys (WebAuthn)

Login sin contraseña con llaves de hardware o biometría del dispositivo.

Dispositivos de confianza

Fingerprinting de dispositivo en cada login para detectar accesos anómalos.

Eventos de seguridad

Registro de logins, fallos MFA, bloqueos IP y actividad sospechosa.

Controles de plataforma

  • WAF con detección de SQLi, XSS, path traversal y command injection
  • Rate limiting por endpoint (login: 10/min, API: 300/min)
  • Protección brute-force: 5 intentos → bloqueo IP 1 hora
  • CSRF global con cookie cb_csrf y header X-CSRF-Token
  • Cookies JWT HttpOnly, SameSite=strict, Secure en producción
  • Headers: HSTS, X-Frame-Options DENY, CSP configurable
  • OpenAPI deshabilitado en producción

Staging Sites

Entorno staging completamente aislado por sitio de producción.

Proceso de creación

  1. Clonar archivos desde producción
  2. Crear base de datos MySQL independiente e importar dump
  3. Desplegar contenedor Docker propio
  4. Configurar dominio staging en Traefik
  5. Emitir certificado SSL (Let's Encrypt)
  6. Activar métricas independientes

Dominios staging

ModoEjemplo (prod: example.com)
subdomainstaging.example.com
randoma1b2c3d4.staging.example.com

Tipos de sincronización

  • files — Solo archivos
  • database — Solo base de datos
  • full — Archivos + base de datos

Direcciones: desde producción (pull) y hacia producción (push).

Seguridad staging

  • Protección por contraseña (basic auth Traefik/Nginx)
  • Restricción por IP (whitelist)
  • Acceso temporal configurable (1–168 horas)
  • Bloqueo de acceso público

Hardening de seguridad

Docker (desarrollo)

  • API y worker sin acceso directo a docker.sock
  • Proxy de socket Docker con permisos mínimos (sin AUTH, SECRETS, SWARM)
  • Contenedores API: no-new-privileges, cap_drop: ALL
  • PostgreSQL y Redis enlazados solo a 127.0.0.1

Instalador (producción)

  • Secretos generados con openssl rand, archivos con permisos 600
  • Firewall: deny incoming por defecto, solo SSH/80/443
  • TLS 1.2+ mínimo en Traefik
  • Sin secretos en logs del instalador
  • Snapshots de rollback automáticos

Secretos que DEBEN rotarse antes de producción

El panel incluye un checklist en Settings → Producción para marcar cada secreto como rotado. Solo Owner y Administrator pueden gestionarlo.

  • APP_SECRET_KEY
  • POSTGRES_PASSWORD y REDIS_PASSWORD
  • REGISTRATION_INVITE_TOKEN
  • Todas las contraseñas *_ADMIN_PASSWORD
  • SUPABASE_JWT_SECRET y POWERDNS_API_KEY

Alertas de recursos (Owner / Administrator)

El módulo de monitoreo evalúa CPU, RAM y disco cada pocos segundos. Si superan el umbral configurado en Settings, se genera una alerta en el panel (campana de notificaciones) y por WebSocket. Umbral por defecto: 90%.

Perfil security (opcional)

docker compose --profile security up -d

Activa CrowdSec (detección de intrusiones) y Fail2ban (bloqueo automático de IPs en la API).

Solución de problemas

La API no responde

docker compose ps api
docker compose logs api --tail=50
curl http://localhost:8000/health

Verificar que PostgreSQL y Redis estén healthy. Comprobar APP_SECRET_KEY y contraseñas en platform.env.

Error de migraciones

docker compose --profile migrate run --rm migrate
docker compose exec postgres psql -U controlbox -c "\dt"

SSL no se emite

  • Verificar que el puerto 80 esté accesible desde internet
  • Confirmar registros DNS propagados (dig api.tudominio.com)
  • Revisar logs de Traefik: docker compose logs traefik
  • El email en CONTROLBOX_ADMIN_EMAIL debe ser válido

WordPress / Staging atascado en "provisioning"

docker compose logs worker --tail=100
docker compose restart worker

Verificar que el worker Celery esté activo y que DOCKER_HOST apunte al proxy de socket.

Permisos insuficientes en el panel

Los permisos se calculan al login combinando roles de identidad + rol de equipo. Tras cambiar roles, el usuario debe cerrar sesión y volver a entrar.

Reparación completa

controlbox repair
sudo bash /opt/controlbox/repair.sh --rollback
Health endpoint GET /health devuelve estado de PostgreSQL y Redis. Útil para balanceadores y monitoreo externo.
